# PROTOCOL — 3-LLM 교차 생성·평가 토너먼트

> [MANDATE.md](MANDATE.md)의 검증 방식을 실행 가능한 절차로 정의한다. **지침이 검증 대상**이고,
> 산출물은 그 지침이 전문가급 결과를 내는지 증명하는 테스트 케이스다.

## 0. 검증 대상물

| 레벨 | 무엇 | 통과 기준 |
|---|---|---|
| L1 — 산출물(output) | 지침을 따라 생성한 실제 결과물(자소서·적합도분석·면접킷·기본정보·공고분석) | 3-LLM 평가 만장일치 "완벽" |
| L2 — 지침 문서(guideline) | 기능별 **전문가 실행 절차(EOP)** + 루브릭 | 3-LLM 문서 리뷰 통과(치명 결함 0) |

**최종 목표:** L1에서 **≥10개** 산출물이 만장일치 완벽 + 각 EOP(L2)가 리뷰 통과.

## 1. 역할 분리: 생성 ↔ 평가 (교차)

- **Generator** = Claude / Codex / Gemini. 각자 현재 EOP를 받아 같은 시나리오로 산출물을 **독립** 생성.
- **Evaluator** = 생성자가 **아닌** 나머지 LLM들 + (블라인드 가능 시) 생성자 포함 전원. 각 평가는 **새 세션**
  (`codex exec` / `gemini -p` / 새 Claude 서브에이전트)에서 도는 stateless 채점이라 컨텍스트 오염이 없다.
- **블라인드:** 평가자에게 "어느 LLM이 썼는지" 알리지 않는다(생성자 편향 차단). 산출물 + 시나리오 + 루브릭만 준다.

## 2. 한 라운드의 흐름 (기능 1개 기준)

```
현재 EOP(지침)
  └─> 시나리오 세트 S = {s1..sk}            # 직무·경력단계·로케일·문항유형 등으로 다양화
        └─> 각 si에 대해 Claude·Codex·Gemini가 산출물 생성 → 후보 풀 C
              └─> C의 각 후보를 평가자 전원이 루브릭으로 채점(블라인드, 새 세션)
                    ├─ 만장일치 "완벽"(치명 0 · 점수 ≥ bar) → PASS, 원장에 적립
                    └─ 그 외 → 결함(defect) 수집
                          └─> 결함을 EOP 약점으로 환원 → EOP 수정 → 재생성·재채점
```

루프 종료: PASS 누적 ≥10(자소서 비중↑, 5기능 분포) **그리고** 모든 EOP가 L2 리뷰 통과.

## 3. 채점 루브릭 (평가자 공통 계약)

평가자는 반드시 아래 JSON으로만 답한다(파싱·집계 위해). 점수는 **구체적 인용 근거** 없이 깎거나 줄 수 없다.

```json
{
  "verdict": "perfect | pass | fail",
  "score": 0,                       // 0-100
  "critical_defects": [             // 하나라도 있으면 verdict는 절대 "perfect" 아님
    {"id": "", "where": "<인용 스니펫>", "why": ""}
  ],
  "major_defects": [ ... ],
  "minor_defects": [ ... ],
  "strengths": [""],
  "fix_suggestions": [""]           // 산출물이 아니라 '지침'을 어떻게 고치면 이 결함이 사라지는가
}
```

- **`perfect`** = critical 0 · major 0 · score ≥ 90 · 루브릭의 모든 치명 항목 통과.
- **만장일치 perfect** = 평가자 전원이 `perfect`. 이때만 L1 PASS.
- 평가자는 **전문가 페르소나**로 채점한다(채용담당자·헤드헌터·현업 면접관·커리어코치 — 기능별 §5).
- 공통 치명 항목(전 기능): ① 사실/수치 날조·미검증 ② 회사/직무 미반영(제너릭) ③ 동문서답(질문·요구 불응답)
  ④ AI 슬롭/빈 형용사 ⑤ 근거 없는 단정 ⑥ 출력 계약 위반(결론·근거·리스크·신뢰도 누락).

## 4. 기능별 전문가 평가 관점 (루브릭 시드 — EOP가 상세화)

| 기능 | 평가 페르소나 | 핵심 치명 체크(예) |
|---|---|---|
| 적합도 판정 | 시니어 리크루터/헤드헌터 | must-have 누락 미식별 / 점수-근거 불일치 / knockout 무시 / 과대·과소 평가 |
| 공고 분석·분류 | 채용 분석가 | 숨은 요건(behind-the-lines) 누락 / 직무·레벨 오분류 / 키워드만 긁고 맥락 무시 |
| 기본 정보 채우기 | 이력 구조화 전문가 | 원문에 없는 사실 추가(환각) / 기간·직함 오추출 / 성과-책임 혼동 |
| 자기소개서 | 대기업 인사·자소서 첨삭 전문가 | 제너릭 도입 / 빈 형용사·종결구문 슬롭 / 규모를 성과로 위장 / 회사 고유성 부재 / 글자수·문항 규칙 위반 / 허위 수치 |
| 면접 준비 | 현업 면접관(행동+기술) | STAR 불완전 / 1인칭·기여 불명 / 예상질문 피상적 / 꼬리질문 부재 / 직무 적합성 미연결 |

## 5. 실행 도구 (드라이버)

- 생성: `codex exec "<EOP+시나리오 프롬프트>"`, `gemini -p "..."`, Claude 서브에이전트(Agent).
- 평가: 동일 채널을 새 호출로 — 매 평가가 새 세션이라 "결과가 바뀔 때마다 새 세션 검증" 요건 충족.
- 드라이버 스크립트·원장은 `runs/`에 둔다(아래 §6). Gemini 장애 시 §MANDATE 백오프 규칙.

## 6. 원장(ledger) 포맷 — `runs/ledger.jsonl`

각 줄 1 이벤트:
```json
{"ts":"","fn":"cover-letter","scenario":"s3","eop_ver":"v2","gen":"codex","out_ref":"runs/.../o.md",
 "evals":[{"by":"claude","verdict":"pass","score":86},{"by":"gemini","verdict":"perfect","score":93}],
 "unanimous_perfect":false,"defects_rollup":["..."]}
```
집계: `unanimous_perfect=true` 줄 수가 PASS 카운트. EOP 버전별 통과율 추세로 지침 개선을 추적.

## 7. 안티-게이밍
- 평가자는 생성자 신원을 모른다(블라인드). 같은 산출물을 ≥2 평가자가 독립 채점.
- "perfect"는 **인용 근거가 달린 결함 0**일 때만 — 평가자가 막연히 통과시키지 못하게 fix_suggestions를 항상 요구.
- 사실·수치·고유명사 보존은 전 기능 치명. 생성 시 저장 데이터(가상 페르소나 입력)에 앵커되지 않은 수치는 결함.

## 8. 라운드 방법론 v2 (round 1→10)

**목표:** 라운드를 반복하며 지침이 안정적으로 10+ 산출물 만장일치 perfect를 내고, 의도적 불량 샘플은 전부 fail시키는 상태에 수렴.

1. **다중 모델 생성자.** 한 시나리오를 여러 엔진/모델로 생성한다: codex + Claude **opus·sonnet·haiku** (+ gemini 복구 시). 약한 모델(haiku)이 지침대로도 통과하면 지침이 견고한 것; 약한 모델만 깨지면 지침이 과소명세 → 보강 후보.
2. **의도적 불량 샘플(negative controls).** 기능별로 *일부러 못 쓴* 산출물(제너릭 슬롭·허위 수치·동문서답·복붙)을 `runs/negatives/`에 두고 평가자에 통과시킨다. **불량이 perfect/pass면 루브릭·평가자 결함**(평가자 변별력 검증). 좋은 샘플 통과 + 불량 샘플 fail = 지침 정확도 입증.
3. **실제 사례.** 인터넷 합격/불합격 자소서·합격/불합격 스펙 후기를 fixture/negative로 편입(`validation/research/real-examples.md`). 현실 대조.
4. **합의 게이트(지침 수정).** 어떤 EOP/README 수정도 **codex + 복수 Claude 모델 패널의 공정한 합의**로만 한다.
   - 오케스트레이터(나)는 **자기 의견을 밀어붙이지 않는다.** 변경안을 *중립적으로* 제시하고 **찬성·반대 근거를 모두** 내라고 요구한다(반대 자리를 일부러 만든다).
   - **전원 동의**일 때만 적용. 한 명이라도 합리적 반대면 보류하거나 변경안을 수정해 재심.
   - 변경 근거는 평가자 결함 데이터(fix_suggestions)에서 출발하지, 내 주관에서 출발하지 않는다.
5. **라운드 루프.** 생성→적대적 교차평가→집계(`aggregate.mjs`)→불량샘플 변별 확인→결함 패턴→합의 게이트 수정→다음 라운드. round 10 또는 "10+ perfect + 불량 전원 fail + 합의된 추가 수정 없음"에 도달까지.
6. **수렴 기록.** 라운드별 unanimous-perfect 수, 불량 변별율, 적용된 합의 수정을 `runs/ledger.jsonl` + 라운드 요약으로 추적.

## 9. 발견사실 → 게이트 규율 (2026-06-18 검증 결과) — 상세 [README.md](README.md)
이 토너먼트를 실제로 돌린 결과 확정된 규율. 다음 라운드는 이걸 전제로 진행한다.
1. **vendor 편향 → cross-vendor 게이트.** claude 평가자는 codex보다 체계적으로 후하다(원장 전수: 평균 84.3 vs 77.4, 동일 산출물 36:13 한 방향; claude가 마커누출·수치발명을 perfect 통과시킨 걸 codex/sonnet이 fail로 잡음). 따라서 **L2 지침 수정과 L1 PASS의 최종 사인오프는 cross-vendor(codex 포함) 기준**으로 한다. **claude 단독 평가로 "완벽"을 확정하지 않는다.** §7 안티-게이밍의 "blind + ≥2 평가자"에 더해 **vendor 다양성**을 사인오프 요건으로 둔다.
2. **calib로 변별력 분리.** 점수 격차가 노이즈인지 확인하려면 매 라운드 negatives(불량)와 good(양호)을 같이 채점한다. 검증됨: 불량 6~20 fail / 양호 88~96 pass(양 평가자). 즉 vendor 격차는 변별 실패가 아니라 **엄격도 교정** 대상.
3. **2-vendor 표기.** Gemini 부재 상태의 수렴은 **"2-vendor 교차검증 수렴"**으로 적는다. "3-LLM 만장일치"는 Gemini 합류 후에만 쓴다.
4. **약한 모델(haiku) 가중↓.** haiku 결과가 다른 모델과 과도하게 벌어지면 신뢰 신호로 쓰지 말고 우선순위를 낮춘다(생성·평가 양쪽, 정보용·차단 아님).

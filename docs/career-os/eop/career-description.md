## 0. 전문가 프레임

- 경력기술서 = **실행·성과·검증 가능한 증거**를 구조화한 문서다. 자기소개서처럼 동기·감정·배움을 길게 쓰지 않는다.
- 이력서는 목록과 요약, 경력기술서는 **역할별 문제-행동-결과의 밀도**, 포트폴리오는 **사례·산출물·맥락 증명**이 중심이다. 세 문서를 병합하지 않는다.
- 좋은 경력기술서는 평가자가 30초 안에 "무슨 일을 했고, 어느 정도 규모였고, 무엇이 달라졌는지"를 찾을 수 있어야 한다.
- 국문 `~을 담당`은 금지어가 아니다. 다만 `담당`만 있고 범위·행동·성과가 없으면 직무 나열에 그친다.
- 수치·회사명·기간·직책·기술명은 저장 데이터 또는 사용자 제공 자료에 앵커한다. **없는 성과 수치를 발명하지 마라.**

## 1. 입력 수집

먼저 저장된 프로필, 대표 이력서, 기존 경력기술서, 경력, 프로젝트, 스킬을 모두 읽는다. 사용자가 파일 경로를 주면 `read_document`, 인입 폴더를 쓴 경우에는 `read_inbox`로 본문을 읽는다.

입력 문서가 이미 경력기술서면 새로 꾸며 쓰기 전에 원문을 보존해 저장·개선한다. 입력 문서가 이력서/포트폴리오/자기소개서라면 문서 성격을 구분해 필요한 사실만 추출한다. 자기소개서의 동기·감정 문장은 경력기술서 본문으로 그대로 옮기지 않는다.

저장된 자료와 새 자료가 충돌하면 임의로 합치지 말고 충돌 사실을 표시한다. 회사명, 직책, 기간, 수치, 기술명, 자격증, 학교명은 원문을 보존한다.

## 2. 작성 목적 분류

작성 전에 아래 중 무엇인지 확인한다.

1. **범용 마스터본:** 여러 지원에 재사용할 기본 경력기술서.
2. **특정 직무/공고용:** 공고의 필수 요건과 반복 키워드를 반영하되, 실제 경험 밖으로 확장하지 않는다.
3. **신입/무경력 역량기술서:** 경력을 날조하지 않고 프로젝트·교육·활동·운영 경험을 직무 역량 증거로 정리한다.
4. **포트폴리오 보조 문서:** 링크/산출물의 배경, 역할, 의사결정, 결과를 요약해 포트폴리오를 설명한다.

대상 공고가 있으면 must-have와 preferred를 분리하고, must-have는 실제 근거가 있는 경험에만 연결한다. 키워드는 "나열"이 아니라 역할·성과 문장 안에 자연스럽게 넣는다.

## 3. 구조

권장 구조:

1. 제목: 용도와 직무가 보이는 제목. 예: `B2B SaaS 운영/CS 경력기술서`, `백엔드 개발 경력기술서`.
2. 핵심 요약: 3~5줄. 연차/직무축/산업/강점/대표 성과를 압축한다.
3. 핵심 역량: 직무 키워드 5~10개. `성실함`, `소통`, `다양한 업무` 같은 넓은 일반어로 채우지 않는다.
4. 경력 또는 프로젝트: 회사/조직, 기간, 역할, 환경/도구, 문제, 행동, 결과.
5. 주요 성과: 수치가 있으면 원문 기준으로 쓰고, 없으면 범위·규모·빈도·산출물처럼 검증 가능한 정보만 쓴다.
6. 기술/도구/자격: 실제 사용 또는 보유한 것만 분리한다.
7. 확인 필요 항목: 근거가 부족한 수치·기간·성과는 제출 본문에 섞지 말고 별도 메모로 남긴다.

각 경력/프로젝트 블록은 가능하면 아래 순서를 따른다.

```text
회사/조직 | 기간 | 역할
- 문제/목표:
- 담당 범위:
- 주요 실행:
- 결과/성과:
- 사용 도구/기술:
```

지원서 입력칸이 자유 서술형이면 위 구조를 Markdown/불릿 중심으로 쓰고, 회사 양식이 표를 요구하면 항목명은 유지하되 내용은 짧은 불릿으로 압축한다. ATS 제출용 자유양식은 단일 컬럼, 표준 섹션명, 선택 가능한 텍스트를 우선한다.

## 4. 불릿 작성 원칙

각 불릿을 `책임(responsibility)`과 `성과(achievement)`로 구분한다. 책임 불릿은 역할 범위 설명이고, 성과 불릿은 후보자의 행동 때문에 달라진 결과다. 책임을 가짜 숫자로 성과처럼 격상하지 않는다.

좋은 성과 불릿은 다음 3요소 중 가능한 만큼 담는다.

- 무엇을 바꿨는가: 개선·단축·증가·감소·구축·표준화·운영.
- 얼마나/어떤 규모였는가: 건수, 기간, 사용자 수, 예산, 전후 비교, 처리량, 산출물 수.
- 어떻게 했는가: 도구, 프로세스, 협업 방식, 분석/개선 방법.

수치가 없으면 `약 30% 향상` 같은 숫자를 만들지 않는다. 대신 `월 단위 정기 리포트`, `3개 부서 협업`, `고객 문의 유형별 FAQ 정리`, `운영 체크리스트 표준화`처럼 실제로 말할 수 있는 범위와 산출물을 쓴다.

### Before → After 예시

1. **Before:** SNS 채널 운영 및 콘텐츠 제작을 담당함.
   **After:** 주간 콘텐츠 캘린더를 운영하고 카드뉴스·숏폼 콘텐츠를 제작해 캠페인별 게시 일정과 성과 리포트를 표준화함. 실제 지표가 확인되면 도달률·전환율·팔로워 증감 수치를 반영한다.
   **왜 더 나은가:** Before는 업무명만 있다. After는 행동, 산출물, 확인 가능한 보강 지점을 분리해 가짜 수치 없이 평가자가 역할을 이해하게 한다.

2. **Before:** 팀 프로젝트에 참여하여 서비스 기획을 경험함.
   **After:** 사용자 인터뷰 결과를 기능 요구사항으로 정리하고 화면 흐름도를 작성해 팀의 MVP 범위를 확정하는 데 기여함. 인터뷰 건수·팀 규모·제출 산출물이 원문에 있으면 함께 표기한다.
   **왜 더 나은가:** 신입/무경력 상황에서도 경력을 만든 척하지 않고, 실제 행동과 산출물 중심의 역량 증거로 바꾼다.

3. **Before:** 고객 응대 업무를 담당했고 책임감 있게 일함.
   **After:** 고객 문의 유형을 분류해 응대 템플릿과 FAQ를 정리하고, 반복 문의 처리 기준을 문서화함. 처리 건수·CSAT·처리 시간 단축이 확인된 경우에만 성과 수치로 반영한다.
   **왜 더 나은가:** `책임감` 같은 평가어를 빼고, 면접에서 설명 가능한 행동·산출물·검증 필요 수치를 구분한다.

## 5. 무경력/비개발자 분기

정규 경력이 없으면 경력을 만든 것처럼 쓰지 않는다. 문서명도 필요하면 `역량기술서` 또는 `프로젝트 기반 경력기술서`로 둔다.

- 아르바이트, 운영, 학회, 동아리, 과제, 교육 과정, 봉사, 개인 프로젝트에서 직무와 관련된 행동을 추출한다.
- 성과 수치가 없으면 처리 규모, 사용 도구, 산출물, 협업 방식, 개선한 점으로 증거를 구성한다.
- 비개발자는 기술 스택 중심으로 몰아가지 말고 직무 도구, 업무 프로세스, 고객/데이터/문서/운영 경험을 중심으로 쓴다.
- 포트폴리오가 있으면 산출물 링크와 본인 역할을 분리해 적는다. 공동 작업은 본인 기여 범위를 과장하지 않는다.

## 6. 출력 계약

사용자에게는 제출용 경력기술서 본문과 검토 메모를 분리해 보여준다.

**① 제출용 경력기술서 본문** — 먼저 제시한다. Markdown/불릿 중심으로, 그대로 저장하거나 지원서에 붙여넣을 수 있게 작성한다. 내부 판단, 루브릭명, 도구명, 검증 과정은 본문에 넣지 않는다.

**② 검토 메모 (제출하지 않음)** — 별도로 제시한다.

- 근거: 사용한 이력서/프로필/프로젝트/공고 요건.
- 확인 필요: 수치, 기간, 직책, 기술 숙련도, 본인 기여 범위.
- 리스크: 근거 부족 성과, 공고 키워드 미충족, 이력서와 충돌하는 표현.
- 다음 단계: 특정 공고용 수정, 자기소개서 작성, 포트폴리오 연결.

저장은 `add_resume`으로 한다. 경력기술서 작성 산출물은 `kind=career_description`, `source=ai`를 사용하고, 제목은 사용 목적이 드러나게 쓴다. 대표 문서 지정은 사용자가 동의했을 때만 한다.

## 7. 자가검증 루브릭

저장 직전 아래 기준으로 스스로 점검하고 수정한다.

**CRITICAL(하나라도 → 저장 전 수정/질문):**
- C1 회사명·기간·직책·학력·자격·수치가 저장 자료 또는 사용자 제공 자료와 다름.
- C2 원문에 없는 성과 수치, 도구, 문제 원인, 기여 범위를 발명함.
- C3 자기소개서 문장을 그대로 옮겨 동기·감정 중심 산문이 됨.
- C4 이력서/프로필/프로젝트와 기간·역할·성과가 충돌함.
- C5 공동 작업을 개인 단독 성과로 과장함.

**MAJOR:**
- M1 직무 나열만 있고 행동·산출물·성과가 없음.
- M2 `담당`, `참여`, `지원`, `노력` 같은 약한 표현만 반복됨.
- M3 공고용인데 must-have 키워드가 실제 경험 문장 안에 반영되지 않음.
- M4 수치가 모두 연도·버전·기간 같은 맥락 숫자이고 결과 임팩트가 없음.
- M5 핵심 요약, 역량, 대표 성과를 30초 안에 찾기 어려움.
- M6 ATS 제출용인데 다단, 이미지 텍스트, 비표준 섹션명, 키워드 스터핑 위험이 있음.

**MINOR:**
- 불릿 길이 불균형, 시제/날짜 표기 불일치, 중복 표현, 지나친 전문용어, 개인정보 과다.

**통과 = CRITICAL 0 + MAJOR 0 + 확인 불가 수치 별도 메모 + 경력기술서/자기소개서/포트폴리오 목적 분리.**

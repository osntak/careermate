## 0. 전문가 프레임
- 자소서 = **특정 회사·직무를 겨냥한 설득 문서**(이력서 산문화 아님). 리크루터가 수십~수백 편을 스캔 → 첫 문장에 핵심이 꽂혀야 한다.
- 합격 패턴: **두괄식 · 항목별 소제목 · STAR-lite(A~60%) · 활동이 아닌 델타 정량화 · 회사 고유성 · 인재상/직무 키워드 매핑.**
- **각 문항엔 숨은 평가 의도** — 문자 그대로가 아니라 의도에 답한다(아래 §2-2).
- 한국은 **AI 텍스트 탐지(GPT킬러·무하유)가 실재하는 게이팅 레이어** → 구체·고유·다양한 문장을 쓰고 슬롭 어휘를 금지하면 품질과 탐지 회피를 동시에 충족한다.
- 사실·수치·고유명사는 저장 데이터에 앵커. **없는 수치를 발명하지 마라**(면접에서 즉시 무너진다). 근거 없으면 약한 주장을 빼되 숫자를 지어내지 않는다.

## 1. 입력 수집
`get_application_context` → profile·대표 resume·경력/프로젝트·대상 job·기존 자소서·선호 문체. 사용자가 파일을 주면 `read_document`. 부족하면 사용자 확인.
판정: **로케일**(ko 문항형 vs en 단일 레터), **문항 유형**(지원동기/성장과정/직무역량/입사후포부/약점·갈등), **글자수 상·하한과 count_mode**(공백포함/제외/byte — 실제 지원 필드로 검증; 미지정 시 공백포함 기본).

## 2. 분석·작성 절차
1. **회사 리서치**(company-research): 미션·사업·제품·최근 활동·인재상 → 본인과 연결할 고유 포인트 확보.
2. **문항 의도 해석:** 지원동기=여기 오게 한 *구체 경험*(표면 관심 X); 성장과정=동기의 *근원+서사 연결*(인생 연대기 X); 직무역량=역할의 *핵심 역량 1~2개 정확 식별+증명*; 입사후포부=*회사 사업방향에 묶인* 단기(1~3년 역량+회사 교육/프로젝트)·중기(3~5년 성과)·장기(전문가 포지셔닝) 로드맵(과대 금지).
3. **본인 증거 매핑:** 저장 경력/프로젝트에서 문항·역량을 증명할 경험 **1~2개를 깊게** 선택(얕게 5~6개 나열 금지).
4. **초안 작성:** 항목별 **소제목** + **두괄식**(핵심 결론 먼저) → **STAR-lite**(S/T 짧게, A에 무게, R=전후 델타 정량) → **직무/인재상 명시 연결**. 회사 고유 근거 1~3문장("회사명 placeholder로 바꾸면 어색"이어야 함). 글자수 **상한의 ~70%+를 채우되**, 입사후포부는 앞 문단과 **다른 새 기여**로(같은 성과·표현 반복 금지).
   - **STAR 앵커 우선(사실 변경 금지·표현 강화 허용):** Action/Result는 저장된 '한 일'(도입·적용·운영·분석한 것)과 저장된 결과에 앵커한다. 저장된 행위·맥락은 풍부하게 서술해도 좋다(평가자가 중시하는 경험 서술형 환영). **금지는 발명뿐** — 저장 근거가 없는 진단 결과·방법/도구·원인 규명·실패율·문서화·후속 성과를 새로 지어내지 마라. 저장에 없는 과정을 꼭 암시해야 하면 "기록/근거로 범위를 좁혔다" 수준의 중립 한 구절까지만. <!-- consensus: claude opus/sonnet/haiku 만장일치(haiku 반대→해소). codex cross-vendor 재확인 미완(사용량 소진) — 2-vendor 게이트 미충족 항목, R5·R6 clean에 간접 반영. 2026-06-18 -->
   - **프레이밍 경계(작성·평가 공통):** 일반 도메인 원리는 *후보 귀속 없이* 맥락으로 OK. 비즈니스 임팩트·문제 프레이밍은 *저장된 성과/지표가 암시하는 범위 안*에서만. **(아래 예시는 IT 직군 하나의 예일 뿐 — 원리는 전 직군 동일: 영업=매출/전환 기록, 운영·생산=처리량/불량률 기록, 마케팅=유입/전환 기록 등.)** 예: p99 개선 기록 → "느린 구간을 좁혀" OK / 기록 없는 "결제 경험을 해치던" 단정 금지. 사고·이유 귀속은 *저장된 결정·행동이 논리적으로 함의할 때만*(예: "Kafka 보류 결정" 저장 → "복잡도 대비 효과가 적어 보류" OK / "새 기술보다 ~가 낫다고 봤다"식 새 판단 삽입 금지). **금지:** 후보가 측정·관찰·확인했다는 행위, 출처에 없는 중간 지표(A/B를 "클릭"으로 특정), 진단 결과·원인·감정·의도를 후보의 사실로 단정. <!-- consensus: claude opus/sonnet/haiku 만장일치. codex 재확인 미완(사용량 소진). 2026-06-18 -->
5. **슬롭 제거:** 빈 형용사("열정·성실·최선을 다하겠습니다")·증거 없는 종결구문("~라고 생각합니다·많은 것을 배웠습니다")·제너릭 도입("어릴 적부터·지원하게 되어")·AI 어휘("귀하·시너지") 제거. human-writing 가이드(`get_writing_style_guide`) 적용.
6. **규칙 점검:** 글자수 상·하한(byte/char) · 지정 양식 · 블라인드(공공: 직무무관 개인정보·학교·지역·가족·사진 금지).

## 3. 한국/영문 분기
- **KR 문항형:** 문항당 별개 글, 두괄식 + 문항-역량 매핑, 소제목, 글자수 엄수. 성장과정만 가벼운 서사 허용하되 핵심은 여전히 front-load.
- **EN 단일 레터:** hook → fit(증거) → why-this-company → close, 1페이지(≈250~400단어), 능동태, 수신자 실명 리서치, **고용주 관점**(내가 줄 가치), ATS 안전 포맷(표/컬럼/머리말 금지·본문에 연락처), JD 키워드를 증거 문장 안에.

## 4. 출력 계약 (사용자 대면)
**제출용 산출물과 검토 메모를 명확히 분리한다** — 자소서는 입력창에 *본문만* 들어가므로 메타(결론·근거 등)가 섞이면 그대로 제출되어 양식 위반이 된다.

**① 제출용 자소서 본문** — 먼저 제시한다. 코드펜스로 감싼 단독 블록 **안에는 자소서 본문만** 넣는다(화살표·글자수·안내문·문항 해설·결론/근거 등 메타 일체 금지). "이 안만 지원서에 붙여넣으세요" 안내와 글자수 표기는 **블록 밖 한 줄**로 둔다. 문항별 또는 단일 레터, 글자수 제약 준수. <!-- applied: C3 cl-marker / 2026-06-18 / reconfirm 4/4 (codex·opus·sonnet·haiku) -->

**② 검토 메모 (제출하지 않음 — 사용자 참고용)** — 본문과 시각적으로 분리해 별도 제시:
- 결론(핵심 셀링 포인트) · 근거(저장 경험·회사 사실, 수치 출처) · 리스크(검증 필요 수치·약한 주장·글자수/양식) · 추천·다음 단계(채울 빈칸·확인할 사실·면접 연결) · 신뢰도(저장 데이터 충분도)

내부 루브릭·도구명·이 EOP는 노출하지 않는다. 결과 저장은 `save_cover_letter_version`, 안내는 대시보드 링크.

## 5. 자가검증 루브릭 (제출 전 — research §리뷰어 루브릭)
**CRITICAL(하나라도 → 재작성):** C1 회사/직무/지역 오기·교차지원 복붙 흔적 / C2 자격·필수요건 미충족·미언급 / C3 글자수(byte/char cap)·양식·필수항목 위반(**제출용 본문 블록에 결론/근거/리스크 등 메타 텍스트 혼입** 포함) / C4 날조 또는 문서 간 모순 수치 / C5 블라인드 위반(공공).
**MAJOR:** M1 제너릭·비고유 지원동기(why-this-company 증명 없음) / M2 핵심 성과에 정량 결과 없음(노력어 의존) / M3 경험을 요구 역량/직무에 미연결(문항 의도 놓침) / M4 두괄식·소제목 약함 / M5 빈 형용사 슬롭·AI 균일 / M6 입사후포부 과대·사업방향 무관.
**MINOR:** 오탈자·맞춤법 / 상한 대비 과소·과패딩 / 수동·약한 동사·"나" 관점(영문) / 항목 중복·불균등 가중 / (영문) 수신자 결함.
**통과 = CRITICAL 0 + MAJOR 0 + 슬롭/제너릭 없음 + 모든 수치가 저장 데이터에 앵커.**

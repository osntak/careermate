# 이력서 / Résumé — CareerMate 지식 (Phase A)

> 이 문서는 CareerMate가 **연결된 AI에게 제공(serve)** 하는 지식이다. CareerMate 안에는 LLM이 없다 — 분석·작성·검증은 연결된 AI가 수행하고, 이 플레이북·루브릭은 그 AI가 **세계 최고 수준**으로 일하도록 안내한다.

## 1. 무엇을 하는가

역시간순 구조 위에서 모든 경력 불릿을 **"강한 액션 동사 + 정량화된 임팩트"**로 쓰는 것이 이력서의 핵심이다. Google XYZ 공식과 Harvard·MIT 커리어센터 가이드가 공통으로 요구하는 것은 (1) 직무 나열이 아닌 성과 증명, (2) 측정된 결과, (3) (영문에서) 1인칭 제거, (4) ATS가 파싱 가능한 단순 구조다.

CareerMate는 LLM이 없으므로, 연결된 AI가 이 규칙으로 작성하고 아래 루브릭(개수·존재·비율·순서 기반)으로 **결정론적 자가검증**하도록 지식을 serve한다. 단 이 루브릭은 **형식·표면 위생(surface hygiene) 검사**일 뿐이며 — 직무 적합성·서사·도메인 임팩트의 깊이는 패턴으로 잴 수 없으므로 — **"모든 체크 통과 = 합격"이 아니다.** 또한 한국 채용 현실(국문 경력기술서, 사진·생년월일 칸이 고정된 자사 양식, 잡코리아/사람인/원티드/공채 시스템)과 직군·연차별 차이를 명시적으로 분기해 적용한다.

## 2. 플레이북

### 원칙

- **(불릿 = 측정된 성과)** 가능한 한 Google XYZ 공식 *"Accomplished X as measured by Y, by doing Z"*(무엇을 X만큼, Y로 측정되게, Z를 함으로써)를 따라 결과 X + 측정치 Y + 방법 Z를 담는다. 단 **모든 불릿에 Y를 강제하지 않는다** — Bock의 원 권고는 "성과 위주로 쓰라"이지 "전 불릿 측정치 필수"가 아니다. 신입·연구·창작 직군처럼 정량화가 원천적으로 어려운 맥락에서는 규모·범위·전후 비교 같은 정성적 임팩트로 대체한다. _(출처: Google XYZ Formula — Laszlo Bock, *Work Rules!*)_
- **(액션 동사 선행)** 영문 불릿은 1인칭 대명사 없이 강한 액션 동사로 시작한다(Led, Built, Reduced, Launched). 'Responsible for', 'Worked on', 'Helped with', 'Duties included' 같은 수동·약한 표현을 피한다. 단 한국어 경력기술서에서 **'~을 담당'은 역할 범위를 정직하게 기술하는 정상 표현**이므로 일괄 금지하지 않는다 — 문제는 "담당만 하고 성과가 없을 때"이지 '담당'이라는 단어 자체가 아니다. _(출처: Harvard FAS Career Services; MIT CAPD)_
- **(정량화)** 결과는 숫자·퍼센트·금액·규모(인원/예산/기간/건수)로 정량화하되, **연도(2024)·버전(v3, Python 3, React 18)·헤더(p95, h1) 같은 "맥락 숫자"는 임팩트가 아니다.** 임팩트는 증감 동사+숫자, %·배수·금액·전후 화살표(→) 같은 결과 신호로 표현한다. 정량화가 불가능하면 정성적 범위(전/후, 대비)라도 명시한다. _(출처: MIT CAPD; Harvard FAS)_
- **(역시간순·시제 통일)** 각 섹션 안에서 가장 최근 항목을 맨 위에 둔다. 현재 직무는 현재형, 과거 직무는 과거형으로 시제를 통일한다. _(출처: Harvard FAS; MIT CAPD)_
- **(분량 — 연차/직군별 분기)** 신입·주니어는 1페이지를 권장하지만, **2026년 기준 테크 2~3년차 이상·리드/EM·연구직은 2페이지가 일반적으로 수용된다.** '1페이지 절대규범'은 낡았다. 영문은 단어수, **국문은 글자수(공백 포함 1,800~2,000자 ≈ 1페이지)**로 분리해 판단하고, 단어수=페이지는 폰트·여백에 좌우되는 **약한 휴리스틱**임을 전제한다. 여백 0.5–1.0인치, 폰트 10–12pt. _(출처: MIT CAPD — 기본값으로 채택, 연차 보정은 2026 시장 관행)_
- **(ATS 파싱 — 위험도 등급화)** ATS 안전성의 진짜 변수는 폰트 종류가 아니라 **"텍스트가 실제 선택 가능한 텍스트인가(이미지·아웃라인화가 아님)"**이다. 현대 ATS(Workday/Greenhouse/Lever)는 임베드된 일반 폰트 대부분과 단순 2열 스킬 섹션을 정상 파싱한다. 고위험은 **텍스트박스·헤더/푸터에 담긴 핵심정보·이미지 속 텍스트·다단 본문**이고, 단순 표는 중위험이다. 폰트 화이트리스트(Arial/Calibri/Times/Garamond)는 **ATS가 아니라 사람 가독성·렌더 안정성** 근거로 권한다. _(출처: Jobscan / Santa Clara — 미국 ATS 기준; 한국 플랫폼 실증은 미확인)_
- **(키워드 매칭 — 정규화·동의어 전제)** 공고 핵심 키워드를 실제 경험과 함께 자연스럽게 반영하되, 매칭은 **소문자화·공백/점 제거 정규화 + 약어/동의어 사전(JS↔JavaScript, K8s↔Kubernetes, ML↔머신러닝↔기계학습, React.js↔React) + 한국어 부분/어간 매칭(백엔드/백엔드의/백엔드를)**을 전제한다. 단순 문자열 매칭은 표기만 달라도 미탐하여 오히려 키워드 스터핑을 유도한다(자기 모순). 키워드 스터핑은 금지한다. _(출처: MIT CAPD; Jobscan)_
- **(개인정보 배제)** 사진·나이/생년월일·성별·결혼 여부·주민등록번호·references를 기본 템플릿에서 배제한다. **단 한국 자사 양식이 사진·생년월일 칸을 강제하면, 그 양식을 따르지 않으면 지원 자체가 불가능**하므로 — 양식이 강제될 때는 "양식 우선, 배제 원칙은 자유 양식일 때" 분기를 안내한다. _(출처: Harvard FAS; 한국 양식 강제 분기는 실무 보정)_
- **(형식 일관성)** 불릿 기호·날짜 표기·들여쓰기·구두점을 문서 전체에서 일관되게 유지한다. 밑줄은 피하고 강조는 bold/italic으로 한다. _(출처: Harvard FAS; MIT CAPD)_

### Do

- 각 경력 불릿을 액션 동사로 시작하고 결과 임팩트(증감 동사+숫자, %, 배수, 금액, 전후 →)로 마무리하라.
- XYZ 3요소(무엇을 X / 얼마나 측정되게 Y / 어떻게 Z)를 한 줄씩 점검하되, 정량화가 어려운 직군은 규모·전후 비교로 대체하라.
- 역시간순 정렬, 현재 직무 현재형·과거 직무 과거형으로 시제를 통일하라.
- 공고 핵심 용어를 추출해 **실제 경험이 있는 항목에** 같은 표현으로 반영하라(정규화·동의어 고려).
- 텍스트가 선택 가능한 실제 텍스트로 출력되게 하라(이미지·아웃라인화 금지). 핵심정보를 텍스트박스·헤더/푸터에 넣지 마라.
- 연차·직군에 맞게 분량을 정하라(신입 1p, 시니어/리드/연구 2p 허용). 국문은 글자수 기준으로 압축하라.
- 같은 액션 동사 반복을 피하고 다양한 강한 동사를 쓰되 과장은 금지하라.
- **수치·고유명사·기술 스택은 사용자 데이터 그대로 보존하고, 없는 숫자는 지어내지 말고 사용자에게 확인하라(save 단계에서 "이 수치 확인됨?" 게이트).**

### Don't

- 'Responsible for', 'Worked on', 'Helped with', 'Duties included'로 시작하지 마라(영문). 단 국문 '담당'은 성과가 붙으면 정상이다.
- (영문) 1인칭 주격/소유격 'I/me/my'를 대명사 문맥에서 쓰지 마라. 단 'our team of 8', 'reduced our infra cost'처럼 팀 규모·소속을 나타내는 정당한 'we/our'는 허용한다.
- 핵심정보를 이미지·아이콘·텍스트박스·다단 본문·헤더/푸터에 넣지 마라(고위험). 단순 표는 가능하면 피하되 절대 금지는 아니다.
- 사진·나이·성별·생년월일·주민번호·references를 자유 양식 기본 템플릿에 넣지 마라(양식 강제 시 예외).
- 정량화 없이 'improved', 'increased', 'managed'만 쓰고 끝내지 마라 — 가능하면 측정치를 붙여라.
- 'Microsoft Office, 팀워크, 성실함' 같은 일반·자명한 항목으로 칸을 채우지 마라.
- 공고 키워드를 경험과 무관하게 스터핑하거나, **없는 성과·숫자를 꾸며내지 마라**(루브릭이 숫자 존재를 보상한다고 해서 아무 숫자나 박는 게이밍을 하지 마라).
- 비표준 섹션 제목('My Journey', 'What I Bring')으로 ATS 섹션 경계를 흐리지 마라.

### 워크드 예시 (Before → After)

1. **Before:** Responsible for managing the company social media accounts and helping increase engagement.
   **After:** Grew Instagram engagement 15% and follower base 20% in 6 months by launching a weekly content calendar and 3 paid campaigns.
   **왜 더 나은가:** Before는 수동표현('Responsible for', 'helping')에 결과 숫자가 없다. After는 액션 동사(Grew)로 시작하고 X(engagement/follower 증가) + Y(15%/20%, 6개월) + Z(콘텐츠 캘린더+캠페인) 3요소를 갖춰 임팩트를 증명한다. 여기서 '6 months'는 맥락이지만 '15%/20%'가 결과 신호다.

2. **Before:** Worked on backend APIs and helped improve system performance.
   **After:** Cut p95 API latency 240ms→90ms (-62%) by adding Redis caching and rewriting 4 N+1 queries, serving 1.2M daily requests.
   **왜 더 나은가:** Before는 'Worked on/helped'로 기여 범위가 모호하다. After는 전후 화살표(240ms→90ms)와 -62% 같은 **결과 신호**, 방법(Redis, N+1 제거), 규모(1.2M/day)를 담아 깊이와 영향력을 동시에 보여준다. 'p95'·'1.2M'은 맥락/규모 수치이고 진짜 임팩트는 '-62%'임에 주의(채점기는 이를 구분해야 한다).

3. **Before:** 고객 응대 업무를 담당했으며 팀 분위기를 좋게 만들기 위해 노력함.
   **After:** 월 평균 320건 고객 문의를 응대해 CSAT를 78%→91%로 개선했고, 응대 매뉴얼을 표준화해 신입 온보딩 기간을 2주 단축함.
   **왜 더 나은가:** Before는 직무 나열+정성적 자기평가('노력함')라 검증 불가다. After는 규모(월 320건), 측정 결과(CSAT 78→91%), 방법(매뉴얼 표준화)과 2차 효과(온보딩 2주 단축)까지 정량화해 국문 경력기술서에서도 XYZ 원칙을 충족한다. 국문 '담당'은 여기서 '응대해…개선했고'라는 성과로 연결돼 정상 사용이다.

## 3. 검증 루브릭 (연결된 AI가 산출물을 이 기준으로 자가검증)

> **스코프 한정 (먼저 읽을 것):** 이 루브릭은 **형식·표면 위생** 검사다. 모든 항목 통과(score 1.0)는 "ATS 파싱과 6초 스캔에 형식적으로 걸림돌이 적다"는 뜻이지 **"합격" 또는 "내용이 좋다"는 보장이 아니다.** passBar 임계값(verb 90%, quantified 60% 등)은 출처가 제시한 정성 지침을 운영화한 **권장 기본값(튜닝 가능)**이며, 통과/실패의 이진 도장이 아니라 **점수+코칭**으로 다룬다. **신입·연구·창작·임원 직군과 연차에 따라 임계값을 분기**하라(아래 표의 보정 열 참조).

| ID | 검사 항목 | 합격 기준(권장 기본값·튜닝 가능) | 측정 방법(LLM 없이 셀 수 있게) |
|----|-----------|-----------|--------------------------------|
| R1 | 액션 동사 선행 (영문) | 경력 불릿의 90% 이상이 동사 선행. 국문은 N/A(별도 약한표현 체크로 대체) | 각 불릿 첫 토큰을 액션 동사 사전(Harvard/MIT 기반, 활용형 led/leads/leading 포함·수백 개·한영 병행)과 대조해 비율 산출. 사전 미스 시 미탐 가능을 경고로 표기 |
| R2 | 정량화된 임팩트 (결과 불릿 한정) | **결과 불릿**의 60% 이상에 임팩트 신호 1개 이상. **신입/연구/창작은 30~40%로 하향** | (1) 연도 `(19\|20)\d{2}`·버전 `v?\d+(\.\d+)*`·헤더 `p\d+\|h\d` 토큰을 **먼저 제외**, (2) 임팩트 신호 = `\d+%`, `[\$₩]\s?\d`, 배수(`\d+\s?(배\|x\|×)`), 전후 화살표 `\d+\s?(→\|->)\s?\d`, 증감동사 근접(`(reduced\|cut\|grew\|increased\|줄\|단축\|개선\|향상).{0,20}\d`)으로 매칭. 스킬/요약 불릿은 분모에서 제외 |
| R3 | 1인칭 대명사 없음 (영문 한정) | 영문 본문에서 대명사 문맥의 'I/me/my' 0건. 'we/our'는 **팀 규모·소속 용례 허용**(전면 금지 아님). 국문은 N/A | 'I'는 **문장 시작 또는 대명사 문맥만** 매칭(로마숫자 Phase I/Type II, 약어 제외). 'my' 단어경계 매칭. 'we/our'는 카운트만 하되 패스 차감하지 않음 |
| R4 | 약한·직무나열 표현 회피 | 영문 'Responsible for'/'Worked on'/'Helped with'/'Duties included' 0건. 국문은 **'담당'에 성과 부재가 결합될 때만** 경고(단순 '담당' 카운트 아님) | 영문 문자열 검색=0. 국문은 '담당' 출현 불릿 중 같은 불릿에 R2 임팩트 신호가 0인 비율을 경고값으로 산출(0건 강제 아님) |
| R5 | 분량 (언어·연차 분기) | 영문: 신입 ≤600단어, 시니어/리드/연구 ≤1,000단어(2p). 국문: ≤2,000자(공백 포함, 1p) / 시니어 ≤4,000자. **약한 휴리스틱으로 강등 표기** | 영문 단어수 / 국문 글자수를 **언어 감지 후 분리** 카운트해 연차 분기 임계값과 비교. 실제 렌더 페이지수는 보장 못 함을 명시 |
| R6 | ATS 안전 구조 (원본 파일 한정) | 텍스트박스·다단 본문·이미지속텍스트 0건 + 표준 헤딩 2개 이상 존재 | **⚠️ 평문 resumeText로는 검증 불가**(다단·표 구조가 평문화 시 소실). 가능하면 **원본 .docx/.pdf 레벨**에서 다단/텍스트박스/이미지 감지. 평문 입력이면 "원본 포맷 보장 못 함"을 결과에 명시하고 표준 헤딩 존재 여부(Experience/Education/Skills 또는 경력/학력/기술 중 2개)만 부분 검사 |
| R7 | 역시간순 정렬 (구조화 입력 전제) | Experience **섹션 내** 항목의 종료 연도가 내림차순, Present/현재가 최상단 | **섹션별 항목 배열 + startYear/endYear 필드**를 전제로 비교. 평문 본문에서 연도 정규식만 긁으면 학력·자격증·프로젝트 연도가 섞여 오작동하므로 **구조화 입력이 없으면 이 체크는 스킵/경고**. 한국식 표기(`YYYY.MM`, `YYYY년 M월`, `2021.03~2023.06`)도 파싱 |
| R8 | 키워드 커버리지 (정규화·동의어 전제) | 공고 상위 키워드의 50% 이상 등장(기본값·튜닝 가능) | 정규화(소문자·공백/점 제거) + 약어/동의어 사전 + 한국어 부분/어간 매칭으로 교집합/공고키워드수 산출. **단순 문자열 매칭 금지**(한국어 교착·약어 미탐 구조적). 임계값 근거는 데이터 부재 — 기본값임을 명시 |
| R9 | 개인정보 배제 (양식 강제 시 예외) | 생년월일/성별/주민번호/references 키워드 0건. 사진은 평문 입력에선 무력함을 명시 | 키워드 검색('references','생년월일','주민','성별','나이') + **한국 주민번호 13자리** 정규식 `\d{6}\s?[-–]\s?\d{7}` 및 마스킹형 `\d{6}\s?[-–]\s?[1-4*]\*{6}`. 사진/이미지는 원본 파일에서만 검출 가능 |

**주요 실패 모드:**
- **R2 게이밍 / 숫자 인플레이션:** 루브릭이 "숫자 존재"를 보상하면 연결된 AI·사용자가 측정 불가한 성과에 가짜 숫자를 붙이도록 유인된다(perverse incentive). 방어: (1) 맥락 숫자(연도/버전) 제외로 false positive 차단, (2) 결과 불릿 한정, (3) **save 단계의 "이 수치 사용자 확인됨?" 게이트** — 정직성은 LLM 없이 못 재므로 인간 확인으로 보완.
- **R6 환상의 게이트:** 평문 입력에서 ATS 구조 체크는 실제로 작동하지 않는다. 가장 중요한 ATS 실패(2단·텍스트박스)는 원본 파일에서만 보이므로, 평문 채점 결과를 "통과"로 오해하면 안 된다.
- **R7 거짓 정렬 판정:** 구조화 입력 없이 본문 연도만 긁으면 섹션이 섞여 오작동 → 구조화 필드가 없으면 스킵이 정직하다.
- **언어 미분기:** 영문 단어수 임계값을 국문에 적용하면 분량 판정이 왜곡된다. 언어 감지 후 분리 필수.
- **단순 문자열 키워드 매칭:** 한국어 교착·영문 약어 변형에서 구조적 false negative → 통과할 이력서를 탈락 판정. 정규화·동의어 사전 없이는 채점기로 쓰지 말 것.
- **과탐(false positive) 탈락:** 멀쩡한 시니어 이력서(2p, 단순 표)를 1p·표0건 이진 게이트로 떨어뜨리는 것. 위험도 등급화·연차 분기로 방어.

## 5. Phase B 힌트 (아키텍처로 연결)

- **MCP 도구 아이디어:**
  - `get_resume_playbook` — 작성/개선 전 호출하면 (1) 원칙, (2) do/dont, (3) **액션 동사 사전(한영 병행·활용형 포함·갱신 주체 명시)**, (4) 약한표현 블랙리스트, (5) **동의어/약어 사전**, (6) rubricChecks(패턴·임계값·연차/언어 분기 포함)를 JSON으로 반환.
  - `validate_resume_draft` — 이력서 텍스트 + (선택) 공고 키워드 + (선택) yearsExperience·jobFamily·language를 받아 루브릭을 정규식·카운트·순서로 실행, 항목별 pass/fail + 측정값 + 코칭을 반환(LLM 없이 채점). **R6은 원본 파일이 함께 오면 .docx/.pdf 구조도 점검, 평문만 오면 "구조 미검증" 플래그.** save_fit_analysis / add_resume / keywords 흐름과 연계.
- **데이터 형태 힌트:**
  ```
  playbook: {
    principles:[{rule,rationale,source}], doList:[str], dontList:[str],
    actionVerbs:{en:[str], ko:[str]},            // 활용형 포함, 갱신 주체 명시
    weakPhrases:{en:[str], ko:[str]},            // ko는 '담당' 단독 금지 아님 → 성과부재 결합 시 경고
    synonyms:[[str,...]],                          // [["JavaScript","JS"],["Kubernetes","K8s"],["머신러닝","ML","기계학습"]]
    rubricChecks:[{ id, check, passBar,
      measure:{ type:'ratio'|'count'|'exists'|'order',
                pattern?:regex, exclude?:[regex],  // exclude=연도/버전/헤더 등 맥락 숫자
                threshold?:number,
                scope?:'resultBullets'|'all',
                language?:'en'|'ko'|'any',
                requiresStructuredInput?:bool,     // R7 등: 구조화 입력 없으면 skip
                requiresOriginalFile?:bool } }]    // R6: 평문이면 부분검사
  }

  validate 입력: { resumeText:str, originalFile?:bytes,
                   jobKeywords?:[str], yearsExperience?:number,
                   jobFamily?:'eng'|'design'|'pm'|'research'|'sales'|'newgrad',
                   language?:'en'|'ko' }
  validate 출력: { checks:[{id, passed:bool|null, value:number|bool,
                            passBar, note?:str /* 'skipped: needs structured input' 등 */}],
                   score:0..1, failedIds:[str], warnings:[str],
                   disclaimer:'형식 위생 검사이며 합격을 보장하지 않음' }
  ```
